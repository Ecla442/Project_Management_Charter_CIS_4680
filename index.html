Project Methodology and Plan

Delivery Objective

The project will deliver a complete, launch-ready website. While the initial requirements are fairly predictive with a clear starting roadmap, later stages will benefit from an adaptive approach as priorities evolve.

Method Selection

Given the need for clarity up front and flexibility as work progresses, the project will use a Hybrid, Adaptive–Iterative approach. This supports structured planning while enabling regular feedback and course-correction.

Chosen Method: Adaptive–Iterative (Hybrid)

  • Why: Encourages constructive feedback, accommodates timeline shifts, and fits a week-by-week planner style.
  • What it looks like: Short iterations (sprints) with planned reviews; the website improves in visible increments.
  • Outcome: A complete site, refined through continuous feedback and iteration.

Approaches Considered

Approach Strengths Limitations / Fit
Predictive Clear, rigid structure; easy to plan early phases. Too inflexible as needs change; not the best fit overall.
Hybrid – Incremental Allows adding features over time. Can feel too spontaneous for preferred working style.
Agile (general) Handles stakeholder feedback well. Stakeholder access is limited right now; needs a tailored cadence.
Hybrid – Iterative (Selected) Stage-based cycles, planned feedback, and flexibility. Requires consistent reviews to achieve benefits.

Iteration Structure & Cadence

  • Sprint Length: 1–2 weeks (weekly planner style).
  • Cycle: Plan → Build → Review (demo) → Retrospective → Re-plan.
  • Feedback: Collected every sprint; drives the next set of tasks.
  • Tracking: Week-by-week to-do list; tasks carry clear acceptance criteria.

Stakeholders & Communication

  • Stakeholder Access: Limited peer communication; confirm who will review each sprint.
  • Reviews: Short demo at sprint end to gather constructive criticism.
  • Updates: Weekly progress note: done, next, risks/blockers.

Quality Gates

  • Definition of Ready (DoR): Task has clear scope, acceptance criteria, and known dependencies.
  • Definition of Done (DoD): Works as specified, tested, integrated into the site; for release items, deployed on the host with domain/DNS and back-end settings configured.

Success Criteria

  • Website completed and launched on a secured domain with back-end settings in place.
  • Feedback incorporated iteratively; visible improvement each sprint.
  • Week-by-week plan followed with flexibility for change.

Tool Selection & Project Setup

Step 1 — Tools Explored & Final Choice

Tools explored: Trello, Microsoft Project, Excel/Google Sheets.

Final choice: Trello. A Kanban-style board aligns with my working style and the project’s hybrid/adaptive-iterative methodology. It lets me post continuous updates, reference prior work, and keep a clear view of what’s coming next.

Tool Justification

  • Fits methodology: Kanban supports iterative sprints, frequent feedback, and easy reprioritization.
  • Board markings by week: Lists/sprint lanes make it obvious which work is upcoming.
  • Status visibility: “To Do / In Progress / Done” labels clarify progress at a glance.
  • Usability: Accessible, low learning curve; quick to set up and manage.
  • Collaboration: School subscription already in place; classmates/stakeholders can view progress and contribute.

Cost consideration: Trello is available through an existing school subscription, which simplifies access and collaboration. Other tools (MS Project, Excel/Google Sheets) are viable, but Trello’s Kanban workflow better supports the chosen methodology.

Step 2 — Structure Overview (Trello)

The project is organized in sprints to improve tracking and accommodate change. I am using Sprint 1–5 (or 6), each representing a time-boxed iteration. This supports my adaptive approach: I can plan, execute, gather feedback, and adjust the backlog quickly.

Board Structure

  • Lists (lanes): S1, S2, S3, S4, S5, S6 (as needed).
  • Cards: Individual tasks and features; each card includes a brief description and checklist.
  • Status labels: To Do, In Progress, Done.
  • Timeline markers: Due dates on key cards to support weekly planning and tracking.
  • Feedback points: End of each sprint (review/demo) to collect constructive criticism and refine next steps.

I’ve used Trello before, so setup is straightforward: the board is labeled for execution, and I will follow through by populating tasks per sprint, tracking progress with labels and due dates, and incorporating feedback after each iteration.
